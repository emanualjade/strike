# D2 Mode Reference

Use D2 when the user wants polished diagram-as-code that renders nicely in the
D2 Playground or D2 CLI.

Prefer Mermaid when the user needs GitHub/Markdown-native rendering. Prefer
Structurizr/C4 for durable architecture documentation, DBML for database-first
models, Graphviz DOT for dense dependency graphs, and PlantUML for formal UML.

## Use D2 for

- polished architecture, service, dependency, or data-flow diagrams
- presentation-friendly diagrams that still need editable source
- system maps with clear boundaries and labeled relationships
- a nicer visual companion to a Mermaid, DFD, or IDEF0 explanation

## Rendering links

Use `references/visual-preview-links.md` for the `View this visually` block.

Default link: https://play.d2lang.com/

## Diagram properly in D2

- Make one diagram answer one question. Split runtime, deployment, schema, and
  ownership views instead of crowding them together.
- Preserve source names when known. Mark gaps as `UNKNOWN`, `TODO`, or
  `ASSUMPTION`; do not invent services, queues, tables, vendors, or flows.
- Use stable short keys and readable labels: `api: API Server`.
- Use containers only for real boundaries such as apps, services, domains,
  networks, teams, or deployment zones.
- Label edges with the call, data, event, dependency, or responsibility crossing
  the boundary.
- Keep styling light unless it materially clarifies the diagram.

## Quick basics

```d2
direction: right

user: User
web: Web App
api: API Server
db: PostgreSQL
email: Email Service

user -> web: uses
web -> api: HTTPS request
api -> db: reads/writes
api -> email: sends confirmation
```

## Nodes and labels

```d2
api: API Server
worker: Background Worker
payment_provider: Payment Provider
```

- A node key is the stable reference used by edges and settings.
- A node label is the readable text after `:`.
- Reference the key in edges, not the displayed label.
- Keep keys simple, lowercase, and stable across revisions.

## Edges

```d2
web -> api: POST /checkout
worker -> api: status update
api <-> cache: read-through cache
service_a -- service_b: shared package
webhook <- payment_provider: event delivery
```

- Use `->` for directed flow or dependency.
- Use `<-` only when it reads clearer than reversing the node order.
- Use `<->` for bidirectional relationships.
- Use `--` only when direction truly does not matter.
- Label meaningful edges. Unlabeled arrows usually hide missing context.

## Containers

```d2
direction: right

frontend: Frontend {
  web: Web App
}

backend: Backend {
  api: API Server
  worker: Worker
  db: Database

  api -> db: SQL
  worker -> db: updates
}

frontend.web -> backend.api: HTTPS
```

- Nest nodes inside containers with `{ ... }`.
- Use dot paths such as `frontend.web` when connecting across containers.
- Avoid decorative grouping. A container should mean a real system boundary or
  ownership boundary.

## Direction

- Available directions are `right`, `down`, `left`, and `up`.
- Put `direction: right` near the top for most architecture and dependency
  diagrams.
- Use `direction: down` for step-by-step processes or top-to-bottom pipelines.
- Keep direction global unless intentionally using a renderer/layout that
  supports more specific layout control.

## Labels

- Use nouns for nodes and verbs, protocols, events, or data names for edges.
- Keep labels short enough to scan. Split long explanations into notes outside
  the diagram.
- If a label or key with special characters fails to render, simplify it or wrap
  it in quotes.

## Quality rules

- Keep the graph small enough to scan; split large systems into focused views.
- Show humans, external systems, internal services, data stores, and queues as
  distinct nodes when that distinction matters.
- Do not mix logical architecture, runtime sequence, deployment topology, and
  database schema unless the user explicitly asks for a rough all-in-one map.
- Include error, retry, timeout, webhook, or async paths when they change the
  user's understanding.
- Exclude secrets, credentials, private tokens, and sensitive sample data.
- Validate in the D2 Playground or D2 CLI when practical before presenting.

## Advanced features

Use advanced D2 features only when they clarify the diagram. For deeper syntax,
use the official docs instead of guessing:

- [D2 docs](https://d2lang.com/) for the language, playground, CLI, and tooling.
- [D2 tour](https://d2lang.com/tour/intro) for the official guided syntax tour.
- [Shapes](https://d2lang.com/tour/shapes/),
  [connections](https://d2lang.com/tour/connections/),
  [containers](https://d2lang.com/tour/containers/),
  [styles](https://d2lang.com/tour/style/), and
  [layouts](https://d2lang.com/tour/layouts/) for deeper features such as
  custom shapes, themes, layout engines, Markdown text, classes, SQL tables,
  icons, imports, and animation.
